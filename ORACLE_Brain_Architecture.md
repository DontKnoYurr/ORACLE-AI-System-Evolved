# ORACLE Brain: Proprietary AI System Architecture

## 1. Introduction
This document outlines the comprehensive architecture for building a proprietary AI system, dubbed the "ORACLE Brain," designed for exclusive personal use. The goal is to integrate with the existing ORACLE frontend, transforming it from a data visualization tool into a fully autonomous, local-first predictive intelligence platform. This system will leverage self-hosted components for its core "brain" functions, ensuring complete data privacy and customizability, thereby eliminating reliance on third-party AI services like Claude or OpenAI.

## 2. Core Principles
*   **Proprietary Control**: Every component, from frontend to backend and AI models, will be under the user's direct control.
*   **Local-First Operation**: The system will primarily operate on local hardware, minimizing external dependencies and ensuring data remains on the user's machines.
*   **Privacy by Design**: No user data or intellectual property will be transmitted to third-party AI services.
*   **Modularity**: Components will be designed to be interchangeable and extensible, allowing for future upgrades and customizations.
*   **Knowledge Integration**: The system will be capable of ingesting and reasoning over the user's personal knowledge base (e.g., Obsidian notes).

## 3. System Architecture
The ORACLE Brain system is conceptualized into four distinct, yet interconnected, layers:

### 3.1. Layer 1: The Interface (Frontend)
This layer is the user's primary interaction point with the ORACLE Brain. It is the existing ORACLE dashboard, which will be enhanced to display insights and interact with the backend.

*   **Current State**: The existing ORACLE dashboard (React, Vite, Tailwind CSS) provides data visualization and interaction capabilities.
*   **Enhancements**: 
    *   **Real-time Data Display**: Display predictions, anomalies, and insights generated by the AI engine.
    *   **User Input**: Provide interfaces for users to submit queries, define new 
signals, or refine AI responses.
    *   **Configuration**: Allow users to configure AI models, knowledge sources, and system parameters directly from the UI.
*   **Technology**: React, Vite, Tailwind CSS, Wouter (for routing), TanStack Query (for data fetching).

### 3.2. Layer 2: The Nervous System (Backend API)
This layer acts as the central communication hub, connecting the frontend to the AI engine and the knowledge base. It will be a custom-built API, ensuring proprietary control over data flow and logic.

*   **Functionality**: 
    *   **API Gateway**: Expose secure endpoints for the frontend to interact with.
    *   **Data Orchestration**: Manage data flow between the frontend, AI engine, and knowledge base.
    *   **Task Management**: Handle long-running AI tasks, such as complex analyses or knowledge base indexing.
    *   **Authentication/Authorization**: Implement user authentication to secure access to the proprietary system.
*   **Technology**: Node.js (Express.js or Fastify) or Python (FastAPI). This choice depends on user preference and existing skill sets. For this project, Node.js with Express.js is a good starting point due to its asynchronous nature and large ecosystem.
*   **Key Components**: 
    *   **RESTful API**: For standard data operations.
    *   **WebSockets**: For real-time updates from the AI engine (e.g., progress of a long-running analysis).
    *   **Task Queue**: (e.g., Redis Queue, BullMQ) for asynchronous processing of AI tasks.

### 3.3. Layer 3: The Reasoning Engine (Local LLM)
This is the core "Brain" of the ORACLE system, responsible for generating insights, predictions, and responses. It will utilize self-hosted Large Language Models (LLMs) to maintain privacy and control.

*   **Functionality**: 
    *   **Natural Language Understanding (NLU)**: Process user queries and knowledge base content.
    *   **Generative AI**: Generate text-based responses, summaries, predictions, and analyses.
    *   **Retrieval-Augmented Generation (RAG)**: Integrate with the Knowledge Base to provide context-aware responses.
    *   **Fine-tuning (Optional)**: Ability to fine-tune open-source LLMs with proprietary data for specialized tasks.
*   **Technology**: 
    *   **Ollama**: A platform for running open-source LLMs locally. It simplifies model management and interaction.
    *   **Local LLMs**: Models like Llama 3, Mistral, or DeepSeek, chosen based on performance and resource requirements.
    *   **LangChain/LlamaIndex**: Frameworks for building LLM applications, facilitating RAG, agent creation, and complex prompt orchestration.

### 3.4. Layer 4: The Long-Term Memory (Knowledge Base)
This layer stores and manages the user's proprietary knowledge, serving as the factual foundation for the AI engine. It will be designed for local-first storage and efficient retrieval.

*   **Functionality**: 
    *   **Document Ingestion**: Process various document types (e.g., Markdown, PDF, text files) from specified directories.
    *   **Text Chunking**: Break down large documents into smaller, manageable chunks for efficient retrieval.
    *   **Vector Embedding**: Convert text chunks into numerical representations (vectors) that capture semantic meaning.
    *   **Vector Search**: Quickly retrieve relevant knowledge chunks based on semantic similarity to user queries.
    *   **Obsidian Integration**: Directly ingest Markdown files from an Obsidian vault, preserving internal links and structure.
*   **Technology**: 
    *   **Local Storage**: A designated directory on the user's machine for storing raw Markdown files (e.g., an Obsidian vault).
    *   **Vector Database**: A local vector store (e.g., ChromaDB, FAISS) to store and index document embeddings. This avoids sending data to cloud-based vector databases.
    *   **Embedding Models**: Open-source embedding models (e.g., `all-MiniLM-L6-v2`) run locally via tools like `sentence-transformers` or integrated with Ollama.

## 4. Integration with Existing ORACLE Frontend
To integrate the newly built backend and AI components with the existing ORACLE frontend, the following modifications will be necessary:

1.  **API Client Update**: Modify the frontend's `api-client-react` to fetch data from the new backend API instead of mock data.
2.  **Environment Variables**: Configure frontend environment variables (e.g., `VITE_API_BASE_URL`) to point to the local backend API.
3.  **Real-time Communication**: Implement WebSocket connections in the frontend to receive real-time updates from the backend (e.g., AI task progress).
4.  **New UI Components**: Develop new React components for AI interaction, configuration, and knowledge base management.

## 5. Step-by-Step Implementation Roadmap

### Phase 1: Setup Local Development Environment
*   **Install Node.js/pnpm**: Ensure the local machine has Node.js (v20+) and pnpm installed.
*   **Install Ollama**: Download and install Ollama to run local LLMs.
*   **Clone Repository**: Clone the ORACLE-AI-System-Evolved repository.

### Phase 2: Develop the Nervous System (Backend API)
*   **Project Setup**: Create a new Node.js/Express.js project (e.g., `backend/`).
*   **Basic API Endpoints**: Implement basic REST endpoints for data (e.g., `/signals`, `/predictions`).
*   **Ollama Integration**: Develop a service to interact with the local Ollama instance for LLM inference.
*   **Knowledge Base Integration**: Implement endpoints to trigger knowledge base indexing and retrieval.
*   **Authentication**: Add a simple authentication layer (e.g., API key or local user/pass).

### Phase 3: Build the Long-Term Memory (Knowledge Base)
*   **Document Ingester**: Write a script to read Markdown files from a specified directory (e.g., `~/ObsidianVault/`).
*   **Chunking and Embedding**: Use a Python library (e.g., `langchain`, `sentence-transformers`) to chunk text and generate embeddings.
*   **Vector Database**: Set up a local ChromaDB instance and store the embeddings.
*   **Retrieval Logic**: Implement functions to perform vector similarity search against the knowledge base.

### Phase 4: Integrate Frontend with Backend
*   **Update `api-client-react`**: Modify existing data fetching logic to call the new backend API.
*   **Environment Variables**: Update `VITE_API_BASE_URL` in the frontend configuration.
*   **New UI for AI Interaction**: Add input fields for AI queries and display areas for AI responses.
*   **Real-time Feedback**: Connect frontend components to WebSocket endpoints for live updates.

### Phase 5: Deployment and Local Hosting
*   **Local Server Setup**: Configure the backend to run as a persistent service (e.g., using `pm2` or `systemd`).
*   **Frontend Serving**: Serve the built frontend assets from the backend server or a local web server.
*   **Reverse Proxy (Optional)**: Use Nginx or Caddy to manage local domains and SSL certificates for secure access.

## 6. Future Enhancements
*   **Agentic Capabilities**: Develop AI agents that can autonomously perform tasks based on signals and knowledge.
*   **Multi-modal Input**: Integrate image, audio, or video analysis into the knowledge base and AI reasoning.
*   **Advanced UI**: Create more interactive and customizable visualizations for AI insights.
*   **Automated Data Ingestion**: Set up webhooks or cron jobs to automatically ingest new data or knowledge.
